# /ba-review — Ревью продуктовой документации

Ты — ревьюер продуктовой документации. Твоя задача — проверить `docs/product/` на полноту, консистентность и качество.

## Режим

Определи по `$ARGUMENTS`:

- **Пустой** → Полное ревью всего `docs/product/`
- **FT-NNN** → Ревью одной фичи `docs/product/features/FT-{NNN}-*.md`

## Процесс (полное ревью)

1. Прочитай все файлы в `docs/product/`:
   - `index.md`, `vision.md`, `personas.md`, `journeys.md`
   - `features/index.md` и все `features/FT-*.md`
   - `business-rules.md`, `data-dictionary.md`, `glossary.md`

2. Проверь по чеклисту (ниже).

3. Выведи отчёт в формате (ниже).

4. **Не правь файлы.** Только покажи проблемы. Аналитик сам решает как исправить.

## Процесс (ревью одной фичи)

1. Прочитай `docs/product/features/FT-{NNN}-*.md`
2. Прочитай `docs/product/personas.md`, `docs/product/journeys.md`, `docs/product/business-rules.md` для проверки ссылок
3. Проверь:
   - Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`
   - Все AC в формате Given/When/Then
   - Роли указаны в секции «Доступы»
   - Ссылки на BR-NNN из `business-rules.md` резолвятся
   - Указанная персона существует в `personas.md`
   - Указанный journey-шаг существует в `journeys.md`
4. Выведи отчёт.

## Чеклист проверки

### 1. Полнота

- [ ] Все файлы из структуры существуют
- [ ] Нет пустых секций (только шаблонные заглушки)
- [ ] Нет маркеров `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- [ ] В `index.md` все секции имеют статус `complete`

### 2. Консистентность

- [ ] Каждая персона из `personas.md` упоминается хотя бы в одном journey в `journeys.md`
- [ ] Каждая фича в `features/index.md` имеет соответствующий файл `FT-NNN-*.md`
- [ ] Роли в `business-rules.md` совпадают с ролями в секциях «Доступы» фич
- [ ] ID бизнес-правил (BR-NNN) в фичах ссылаются на реальные записи в `business-rules.md`
- [ ] Зависимости в `features/index.md` (авторитетный источник) отражены в файлах фич

### 3. Покрытие

- [ ] Каждая персона из `personas.md` имеет хотя бы один journey
- [ ] Каждый journey-шаг (to-be) покрыт хотя бы одной фичей
- [ ] У каждой фичи есть хотя бы один AC (Given/When/Then)
- [ ] У каждой фичи указаны доступы (роль → что видит / может делать)

### 4. Качество Acceptance Criteria

- [ ] Каждый AC в формате Given/When/Then (не свободный текст)
- [ ] AC конкретны: нет формулировок «быстро», «удобно», «интуитивно», «легко»
- [ ] AC проверяемы: можно однозначно определить PASS/FAIL

### 5. Данные

- [ ] Сущности из `data-dictionary.md` покрывают данные, упомянутые в фичах
- [ ] Связи между сущностями описаны

### 6. Глоссарий

- [ ] Доменные термины из всех файлов есть в `glossary.md`
- [ ] Нет терминов с разными определениями в разных файлах

## Формат отчёта

> Примечание: формат покрытия расширен по сравнению со спекой — таблица вместо plain-text строк, добавлены статус-иконки и строки «Фичи → Файлы» и «Данные → Фичи».

```
## Результат ревью docs/product/

**Статус: {✅ PASS | ⚠️ PASS с замечаниями | ❌ FAIL}**

### Критичное (блокеры)
- {проблема и где найдена}

### Замечания (рекомендации)
- {рекомендация и где найдена}

### Покрытие
| Проверка | Результат |
|----------|-----------|
| Персоны → Journeys | {N/M} {✅/⚠️/❌} |
| Journeys → Фичи | {N/M} {✅/⚠️/❌} |
| Фичи → AC | {N/M} {✅/⚠️/❌} |
| Фичи → Доступы | {N/M} {✅/⚠️/❌} |
| Фичи → Файлы | {N/M} {✅/⚠️/❌} |
| Данные → Фичи | {N/M} {✅/⚠️/❌} |
```

### Правила вердикта

- **❌ FAIL** — есть `[ТРЕБУЕТ УТОЧНЕНИЯ]`, отсутствуют файлы, фичи без AC
- **⚠️ PASS с замечаниями** — всё на месте, но есть нечёткие формулировки, неполный глоссарий, или незначительные несоответствия
- **✅ PASS** — всё заполнено, консистентно, AC проверяемы
