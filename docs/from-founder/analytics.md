1. Общая суть модели
Rewarble — это сервис, который позволяет пользователю купить или получить voucher/gift card, затем ввести код на стороне Rewarble и конвертировать его в другой цифровой продукт: e-wallet top-up, PayPal top-up, virtual Visa/Mastercard, bank transfer, gift cards, gaming cards, mobile top-ups и другие цифровые услуги. На сайте Rewarble указано, что сервис даёт доступ к 80+ продуктам и сервисам, а также позволяет использовать Rewarble balance для PayPal, bank transfers, gaming cards, mobile top-ups и streaming subscriptions. 
С точки зрения бизнес-процесса Rewarble это конверсионная прослойка:
Деньги / voucher / gift card
→ Rewarble code
→ Rewarble balance / order
→ выбранный цифровой продукт
→ исполнение
Главная ценность — пользователь или партнёр получает универсальный инструмент, когда прямой платёж неудобен, недоступен или ограничен.
2. Где покупается voucher
Публичный flow Rewarble предполагает, что пользователь получает voucher у trusted partner / distributor, а затем redeem’ит его на сайте Rewarble. На странице redeem прямо указано: если у пользователя нет voucher, он может получить его у trusted distributor. 
То есть покупка voucher может происходить не только на самом Rewarble, а через внешних дистрибьюторов и маркетплейсы. Например, Rewarble vouchers доступны на площадках вроде Eneba, SEAGM, Gamecardsdirect, beCHARGE и других реселлеров. На beCHARGE flow описан так: пользователь покупает Rewarble PayPal voucher, переходит на Rewarble redeem page, вводит 16-значный код и PayPal email. 
Для нашего MVP это важный вывод: voucher layer можно отделить от продукта. Нам не обязательно сразу строить большой marketplace. Можно начать с модели, где клиент получает код через партнёра, менеджера, API или ручную операцию, а наша система отвечает за проверку, учёт и исполнение.
3. Как генерируется код
Публично Rewarble показывает механику 16-значного кода. На redeem page пользователь вводит email и 16-digit code. 
Из этого можно вывести базовую механику: каждый voucher — это уникальный код, к которому привязаны номинал, валюта, срок действия, статус и доступный продуктовый сценарий. На стороне системы код должен быть создан заранее или сгенерирован при покупке/выдаче.
Для нашего продукта возможны три модели генерации кодов:
Модель 1 — pre-generated code pool. 
 Коды заранее создаются в системе и передаются партнёрам/клиентам. Это удобно для MVP, потому что можно контролировать количество, номиналы и статусы.
Модель 2 — on-demand generation.
 Код создаётся в момент заказа. Это лучше для B2B/API-сценариев, когда клиент хочет выпускать коды под конкретные суммы.
Модель 3 — supplier-issued codes.
 Коды генерирует внешний поставщик, а наша система только валидирует и исполняет redeem. Это снижает техническую сложность, но создаёт зависимость от поставщика.
Для MVP лучше рассматривать pre-generated или manual/on-demand generation, потому что это проще контролировать и легче аудировать.
4. Где хранится баланс
На странице PayPal Gift Card Rewarble описывает, что при redeem кода средства сначала попадают в Rewarble account, а уже оттуда пользователь выбирает направление: PayPal top-up, bank transfer или prepaid Visa card. 
Это важный продуктовый момент: после ввода кода value не обязательно сразу уходит в конечный продукт. Сначала он может попасть во внутренний баланс / account / ledger. После этого пользователь выбирает, куда направить средства.
Для нашего MVP это означает, что нужен ledger (баланс), даже если мы не делаем полноценный пользовательский кошелёк. Ledger должен фиксировать:
- кто внёс value
- каким кодом
- какой номинал
- какая комиссия
- какой продукт выбран
- сколько списано
- сколько исполнено
- какой остаток, если он допустим
Без ledger невозможно нормально считать маржу, разбирать ошибки, делать reconciliation и защищаться от спорных ситуаций.
5. Как происходит redeem
Публичный redeem-flow у Rewarble выглядит так: пользователь заходит на redeem page, вводит email, вводит 16-значный код, соглашается с Terms and Conditions / Cancellation Policy и запускает redeem. 
Для PayPal top-up на сайте Rewarble описан процесс: купить gift card у партнёра, redeem на Rewarble, выбрать PayPal top-up, указать сумму и PayPal email, подтвердить перевод. 
С точки зрения бизнес-процесса redeem состоит из нескольких шагов:
1. Пользователь вводит код
2. Система проверяет существование кода
3. Система проверяет статус кода
4. Система проверяет срок действия
5. Система проверяет лимиты и риск-флаги
6. Код переводится в статус redeemed / locked
7. Создаётся order на выбранный продукт
8. Система исполняет заказ автоматически или вручную
9. Пользователь получает результат
Критически важно: после ввода кода его нельзя сразу “сжигать” окончательно, если исполнение конечного продукта может упасть. Лучше использовать промежуточный статус: validating → locked → processing → completed
Если исполнение не удалось, нужен сценарий возврата value в баланс или перевод заказа в manual review.
6. Какие продукты доступны
Rewarble публично заявляет несколько категорий продуктов: e-wallet top-ups, virtual debit cards, gifts/rewards, PayPal top-up, bank transfer, prepaid Visa card, gaming cards, mobile top-ups, streaming subscriptions. 
Также реселлеры описывают Rewarble как способ redeem gift card в e-wallet credit для PayPal, Astropay, Papara и других направлений. 
Для нашего MVP не нужно копировать весь ассортимент. Наоборот, широкий каталог увеличит риски: больше поставщиков, больше правил, больше возвратов, больше support cases.
Рациональный MVP должен выбрать 1–2 направления:
1. Самый востребованный продукт у текущих клиентов
2. Самый безопасный продукт с точки зрения фрода и юрки
voucher → internal balance → digital goods / wallet top-up
или
voucher → order → ручное исполнение выбранного цифрового товара
7. Где возникает комиссия
Комиссия в такой модели может возникать в нескольких местах.
Первое место — redemption fee / service fee. Пользователь вводит код и при выборе направления видит комиссию за обработку или конвертацию. На странице fees Rewarble указывает, что комиссии по Visa/Mastercard зависят от выбранной карты, а по bank transfer и другим продуктам применяются отдельные fee structures. 
Второе место — спред между закупкой и продажей value. Например, сервис может покупать цифровой продукт дешевле номинала или иметь партнёрскую скидку, а пользователю отдавать по полной цене или с комиссией.
Третье место — FX / currency conversion. Если voucher в одной валюте, а конечный продукт в другой, сервис может зарабатывать или терять на курсе.
Четвёртое место — B2B fee. В нашем кейсе заявлена комиссия 5–8% с оборота. Нужно обязательно уточнить: это gross revenue или чистая маржа после всех расходов.
Для MVP нужно явно разделить:
- GMV — общий оборот
- Revenue — наша комиссия
- COGS — стоимость закупки продукта / поставщика
- Payment costs — комиссии платёжек
- Fraud reserve — резерв на фрод и возвраты
- Gross margin — то, что остаётся до операционных расходов
- Net margin — то, что остаётся после поддержки, разработки, юрки и бухгалтерии
Если 5–8% — это revenue, проект может быть интересным. Если внутри этих 5–8% сидят поставщики, фрод, возвраты и операционка, маржа может оказаться слабой.
8. Кто несет риск chargeback / fraud
В gift card / voucher бизнесе риск зависит от того, где именно происходит платёж и кто является merchant of record.
Если пользователь покупает voucher у внешнего маркетплейса, то первичный card chargeback может лечь на маркетплейс или продавца voucher. Но если voucher уже был redeemed и value выведен в другой продукт, возникает риск, что исходный платёж будет оспорен, а товар уже невозвратен.
Типовые fraud-сценарии:
- покупка voucher украденной картой
- перепродажа украденных gift cards
- redeem с подозрительных аккаунтов
- мультиаккаунтинг
- использование сервиса для laundering value
- споры по неоказанной услуге
Для нашего MVP нужно считать, что риск на старте в значительной степени будет на нас, если мы контролируем redeem и исполнение. Поэтому нужны:
- лимиты на клиента
- ручная проверка крупных заказов
- hold-период для подозрительных операций
- blacklist клиентов / email / IP / реквизитов
- журнал операций
- fraud reserve
- понятные правила возврата
Особенно опасны продукты, которые невозможно отозвать после выдачи: пополнения, gift cards, prepaid cards, crypto-like value, игровые балансы.
9. Где нужен KYC / AML
Rewarble и похожие сервисы находятся на стыке gift cards, e-money, prepaid value, payouts и digital goods. Поэтому KYC/AML зависит от юрисдикции, суммы, продукта и направления движения value.
На стороне публичного Rewarble пользователь вводит email и код, но для некоторых продуктов, стран, сумм или партнёров могут потребоваться дополнительные проверки. В публичных материалах по связанным voucher/e-wallet сценариям видно, что третьи стороны могут регулировать redemption, account crediting, expiry, fees and limitations. Например, в terms Luxon указано, что prepaid vouchers от third-party providers, включая Mechhive B.V. trading as Rewarble/Skine, регулируются terms issuer’а, а Luxon только facilitates redemption and account crediting. 
Для нашего MVP нужно не пытаться обойти KYC, а заранее определить:
- когда KYC не нужен
- когда нужен light KYC
- когда нужен full KYC
- когда транзакцию запрещаем
Минимальные AML-триггеры:
- крупная сумма
- частые операции
- несколько аккаунтов на одни реквизиты
- подозрительные гео
- несовпадение страны клиента и продукта
- попытка дробления платежей
- повторные failed / refunded операции
На старте лучше ограничить MVP так, чтобы не попадать в тяжёлую regulated-зону: меньше продуктов, меньше стран, лимиты, ручная проверка, B2B-клиенты с предварительной верификацией.
10. Какие лимиты
Публичные страницы Rewarble и реселлеров показывают, что существуют номиналы, сроки действия и product-specific fees. Например, на Eneba указано, что Rewarble Gift Cards имеют expiry 9 months, а transaction fees применяются во время redemption process и могут отличаться по продукту. 
Для нашего MVP лимиты нужно закладывать сразу. Они нужны не только для compliance, но и для защиты маржи.
Минимальные лимиты:
- лимит на один код
- лимит на один заказ
- лимит на клиента в день
- лимит на клиента в месяц
- лимит на продукт
- лимит на страну
- лимит на ручную проверку
Для MVP
до $100 — автоматическая обработка
$100–500 — базовая risk-проверка
$500+ — manual review
новый клиент — сниженные лимиты
проверенный B2B-клиент — индивидуальные лимиты
11. Какие сроки исполнения
Rewarble позиционирует сервис как быстрый. На странице incentives указано, что пользователь redeem’ит voucher, выбирает reward и получает возможность тратить в течение 2 минут. 
В support-разделе по virtual card top-up указано, что большинство транзакций появляются instantly, но возможна короткая задержка, и если баланс не обновился, нужно подождать до 10 минут. 
Для нашего MVP можно не обещать мгновенные выплаты, если процесс ручной или полуавтоматический. Лучше заложить SLA автоматические операции до 5–10 минут, ручная проверка до 1–24 часов
12. Какие статусы заказа нужны
Для Rewarble-like сервиса статусы не просто UX. Это основа контроля денег, фрода и операционки.
Минимальная модель статусов
created — заказ создан
code_entered — пользователь ввёл код
validating — код проверяется
invalid — код не найден / неверный
expired — срок действия истёк
locked — код временно заблокирован под заказ
redeemed — код успешно принят системой
product_selected — пользователь выбрал направление
risk_check — заказ на проверке
manual_review — требуется ручное подтверждение
processing — исполнение запущено
completed — заказ успешно исполнен
failed — исполнение не удалось
refunded — value возвращён
cancelled — заказ отменён
chargeback_flagged — выявлен спор / риск
fraud_blocked — заказ заблокирован по риску
Для MVP можно сократить, но нельзя убирать ключевые:
created
validating
redeemed
processing
completed
failed
manual_review
refunded
fraud_blocked
Особенно важен статус locked. Он защищает от ситуации, когда один и тот же код пытаются использовать дважды или когда пользователь обновляет страницу во время операции.
Итоговая бизнес-схема Rewarble-like процесса
1. Клиент покупает или получает voucher у партнёра / продавца / через B2B-канал
2. Voucher представлен уникальным кодом с номиналом, валютой, сроком действия и статусом
3. Пользователь вводит код на redeem-странице
4. Система проверяет код, лимиты, срок действия и risk-факторы
5. Код блокируется и конвертируется во внутренний value / balance / order
6. Пользователь выбирает конечный продукт
7. Система применяет комиссию и показывает итоговую сумму
8. Заказ проходит автоматическую или ручную проверку
9. Система исполняет продукт через поставщика, API или manual operation
10. Ledger фиксирует всю операцию: вход, комиссию, расход, результат
11. Пользователь получает подтверждение
12. Операция закрывается или уходит в failed / refund / manual review
