1. Цели 
- Выпуск “универсальных” ваучеров (gift codes) партнёрам B2B.
- Конечный пользователь редимит ваучер на сайте и сразу получает конечный цифровой товар (Steam и др.).
- Минимизировать регуляторные риски “stored value” и операционные риски фрода/споров.
Принципы (MUST)
- NO WALLET / NO BALANCE: система не хранит баланс, не ведёт счёт пользователя.
- NO PAYOUT / NO CASH-LIKE: отсутствуют выплаты, выводы, банковские переводы, e-wallet topups, prepaid карты.
- FULL REDEEM ONLY: ваучер редимится только полностью, остаток всегда 0.
- EMAIL-DELIVERY: минимальные данные — email для доставки и статуса; без аккаунтов/паролей.
- FINALITY AFTER DELIVERY: после выдачи конечного кода возвраты/отмена невозможны (кроме явной ошибки системы до выдачи).

1. Роли и области ответственности
Роли
- Партнёр (B2B Seller): закупает ваучеры, продаёт на маркетплейсах, несёт ответственность за B2C-disputes/chargebacks на маркетплейсе.
- Платформа (мы): выпускает ваучеры, принимает B2B оплату, обеспечивает редим и поставку конечных кодов, KYB/AML/KYT контроль партнёров, антифрод на редиме.
- Конечный пользователь: вводит ваучер, выбирает товар, получает код.
Разделение ответственности (MUST)
- Chargeback/dispute конечника на маркетплейсе — ответственность партнёра.
- Мы предоставляем партнёру Proof of Delivery по факту редима/доставки (без PII конечника).

2. Термины и сущности данных
Сущности
- Partner: юрлицо партнёра, KYB статус, risk tier, лимиты, whitelisted crypto wallets.
- VoucherBatch: партия ваучеров (для отчётности/инвойсов).
- Voucher: код ваучера (хранить безопасно: хэш/шифрование), номинал, валюта, срок действия, статус.
- RedemptionOrder: заявка редима (email, выбранные SKU/“корзина”, статус, тайминги, ссылки/токены).
- FulfillmentItem: единица выдачи конечного продукта (Steam key / gift card code / top-up), поставщик, статус.
- DeliveryToken: одноразовый токен страницы доставки (TTL + ротация).
Статусы ваучера (MUST)
- ISSUED → REDEEM_STARTED → (PENDING_FULFILLMENT | DELIVERED | FAILED_BEFORE_DELIVERY | EXPIRED)
- Ваучер считается потраченным (consumed) только при переходе в DELIVERED.
- В PENDING_FULFILLMENT ваучер зарезервирован под заказ и не может быть использован повторно.

3. Продуктовые ограничения (анти “stored value”)
Номиналы и цены (MUST)
- Стартовые номиналы: только “ровные” (например, 5/10/20/50), без копеек.

“Корзина” (MUST/SHOULD)

- MUST: запрещены частичные редимы с остатком (никаких “оставить на потом”).
Запрещённые функции (MUST)
- Перевод/подарок баланса между пользователями.
- Аккаунт с накоплением средств, пополнения, история “баланса”.
- Продукты cash-like.

4. Поток редима (B2C)
4.1. Страница редима 
/redeem
 (MUST)
Поля:
- Voucher code
- Email (обязателен)
- Выбор товара/товаров (каталог)
Логика:
1. Проверка кода: формат, хэш-поиск, статус, срок, лимиты по попыткам.
2. Валидация email (syntax + MX optional).
3. Показ только in-stock SKU (SHOULD) + попытка “мягкой резервации” (SHOULD, TTL 1–3 мин).
4. Подтверждение: создаётся RedemptionOrder, генерируется DeliveryToken.
4.2. Страница доставки 
/delivery/<token>
 (MUST)
Отображает:
- Статус заказа (Pending/Delivered)
- Если Delivered: код(ы) + инструкции использования + кнопки:
  - “Скопировать”
  - “Отправить повторно на email” (rate limit)
- Если Pending: ETA, кнопка “Проверить статус”, ссылка на поддержку.
4.3. Email-уведомления (MUST)
- Order received: “обычно 5–15 минут” + delivery link.
- Delayed: если >15 минут, письмо “задержка, работаем”.
- Delivered: итоговое письмо с кодом(ами) + delivery link.
- Escalated: если >=6 часов — “передано в поддержку” + ticket id.
4.4. SLA Pending (MUST)
- Норма: 5–15 минут.
- Если >15 минут: статус PENDING_EXTENDED, триггер delayed-email.
- Если >=6 часов: автосоздание тикета + статус MANUAL_ESCALATION + обязательная ручная обработка.
4.5. Out-of-stock / сбой поставщика (MUST)
- Если SKU ушёл в OOS до подтверждения — пользователь выбирает альтернативу.
- Если сбой случился после подтверждения:
  - заказ остаётся PENDING_FULFILLMENT
  - пользователю предлагаются альтернативы “того же номинала” на delivery page/email (SHOULD)
  - поддержка может заменить SKU внутри заказа (MUST), сохранив правило: итоговая сумма = номинал, остаток 0.

6. Антифрод и защита от перебора
6.1. Защита кода ваучера (MUST)
- Длинный непредсказуемый код (криптостойкий генератор).
- Хранение: хэш + соль (или шифрование) — не хранить в открытом виде.
- Ограничение попыток:
  - per IP / per subnet
  - per device fingerprint
  - per code
- Progressive delay при ошибках.
6.2. Captcha/step-up (MUST)
- Captcha включается:
  - после N неуспешных попыток,
  - при подозрительном velocity,
  - при high-risk SKU (Steam) на определённых порогах.
- Не включать всегда (чтобы не убивать UX), но MUST иметь авто-включение.
6.3. Детект аномалий (SHOULD)
- Частые неуспешные попытки + разные коды
- Массовые редимы на разные email с одного IP
- Неестественные пики по Steam
- Гео/ASN аномалии (прокси/VPN сигналы)

7.4. Без кредита на старте (MUST)
- Выпуск ваучеров партнёру только по педоплате.

11. Политики, юридические формулировки, доказательства
11.1. Finality (MUST)
- “После выдачи конечного кода операция считается завершённой; возвраты/отмены невозможны”.
11.2. Proof of Delivery (MUST)
- Отчёт/сертификат для партнёра:

11.3. Срок действия ваучеров (MUST)
