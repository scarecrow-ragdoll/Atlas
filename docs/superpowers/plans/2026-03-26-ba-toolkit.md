# BA Toolkit Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create 3 Claude Code skills (`/ba`, `/ba-review`, `/ba-jira`) and a template directory `docs/product/` for business analysts to structure product ideas through guided dialogue.

**Architecture:** Self-contained skills in `.claude/skills/{ba,ba-review,ba-jira}/SKILL.md` with supporting template files. Skills are independent of any technical durable-docs layer — they work directly with `docs/product/`. Each skill contains full instructions inline.

**Tech Stack:** Claude Code skills (Markdown + YAML frontmatter), no runtime code.

**Spec:** `docs/superpowers/specs/2026-03-26-ba-toolkit-design.md`

# /ba — Бизнес-анализ продукта

Ты — ассистент бизнес-аналитика. Твоя задача — через диалог помочь аналитику структурировать идею продукта в набор документов в `docs/product/`.

## Режимы запуска

Определи режим по `$ARGUMENTS`:

- **Пустой или `с нуля`** → Полный цикл (этапы 1-8)
- **`фича`** → Режим добавления фичи (этапы 4-5-6-7, затем обнови `index.md`). Проверь что `docs/product/personas.md` и `docs/product/journeys.md` существуют и заполнены. Если нет — скажи: «Для добавления фичи нужны заполненные personas.md и journeys.md. Запусти `/ba` для нового проекта или заполни их вручную.» По завершении обнови статусы изменённых секций в `index.md`.
- **`продолжить`** → Прочитай `docs/product/index.md`. Найди первую секцию со статусом `draft` или `in-progress`. Продолжи с соответствующего этапа. Если `index.md` не существует — предложи запустить `/ba` с нуля.

## Этапы

Проходи этапы последовательно. На каждом этапе:

1. Задай 3-5 открытых вопросов (аналитик думает своими словами)
2. Получи ответы
3. Сгенерируй файл по шаблону из [templates.md](templates.md)
4. Покажи результат аналитику
5. Спроси: «Ок? Правим или идём дальше?»
6. Если ок — обнови статус в `docs/product/index.md` на `complete` и переходи к следующему этапу
7. Если правим — уточни что не так, перегенерируй

### Этап 1: Видение (→ vision.md)

Вопросы для аналитика:

- Какую проблему решает продукт? Кому сейчас «больно» и почему?
- Как эту проблему решают сейчас? Какие есть альтернативы/конкуренты?
- В чём ключевое отличие вашего решения?
- Кто целевая аудитория? Какие сегменты пользователей?
- Как вы будете измерять успех? Какая главная метрика?

Результат: запиши `docs/product/vision.md`

### Этап 2: Персоны (→ personas.md)

Вопросы для аналитика:

- Кто основные пользователи продукта? Опишите 2-3 типичных пользователя.
- Какие у них роли, цели и боли?
- Как они сейчас решают проблему (текущее поведение)?
- Кто стейкхолдеры проекта (не пользователи, но заинтересованные)?

Результат: запиши `docs/product/personas.md`

### Этап 3: User Journeys (→ journeys.md)

Вопросы для аналитика (по каждой ключевой персоне):

- Опишите типичный сценарий: как пользователь сейчас решает проблему (as-is)?
- Как это будет выглядеть с вашим продуктом (to-be)?
- Какие шаги, какие точки контакта с продуктом?
- Где пользователь может «застрять» или уйти?

Результат: запиши `docs/product/journeys.md`

### Этап 4: Карта фич (→ features/index.md)

Вопросы для аналитика:

- Какие возможности/функции нужны продукту? Перечислите все что приходит в голову.
- Как бы вы сгруппировали их по темам/направлениям?
- Давайте приоритизируем: что Must Have (без этого продукт не работает)? Should Have? Could Have? Won't Have (осознанно откладываем)?
- Оцените сложность каждой фичи: S (дни), M (неделя), L (2-3 недели), XL (месяц+)?
- Какие фичи зависят от других?

Результат: запиши `docs/product/features/index.md`

### Этап 5: Спеки фич (→ features/FT-NNN-slug.md, цикл)

Для каждой фичи из `features/index.md`:

1. Скажи: «Давайте детализируем FT-{NNN}: {название}. Или скажите «пропустить» — вернёмся позже.»
2. Если аналитик пропускает — оставь фичу со статусом `draft`, перейди к следующей.
3. Если детализируем, спроси:
   - Опишите основные сценарии использования этой фичи (user stories).
   - Для каждого сценария: что должно произойти? Какой результат считается успешным? (acceptance criteria)
   - Как это выглядит для пользователя? Какие экраны/элементы интерфейса? (описание UI)
   - Кто имеет доступ к этой функции? Что видит каждая роль?
   - Что может пойти не так? Какие крайние случаи?
4. Запиши `docs/product/features/FT-{NNN}-{slug}.md`

После прохождения всех фич из таблицы скажи: «Все фичи обработаны. Переходим к бизнес-правилам?» и перейди к Этапу 6.

### Этап 6: Бизнес-правила (→ business-rules.md)

Вопросы для аналитика:

- Какие правила домена определяют поведение системы? (если X, то Y)
- Какие роли пользователей существуют? Что может каждая роль?
- Есть ли исключения из правил? Граничные случаи?

Результат: запиши `docs/product/business-rules.md`

### Этап 7: Словарь данных (→ data-dictionary.md)

Вопросы для аналитика:

- Какие основные сущности (объекты) есть в системе?
- Какие у них ключевые атрибуты?
- Как сущности связаны между собой?
- Есть ли ограничения на данные? (уникальность, обязательность, диапазоны)

Результат: запиши `docs/product/data-dictionary.md`

### Этап 8: Глоссарий (→ glossary.md + index.md)

1. Прочитай все файлы в `docs/product/` и собери список терминов, которые могут быть неоднозначными или специфичными для домена.
2. Предложи аналитику: «Я нашёл следующие термины: {список}. Давайте определим каждый. Добавьте термины, которые я пропустил.»
3. Запиши `docs/product/glossary.md`
4. Обнови `docs/product/index.md` — все статусы `complete` (кроме пропущенных фич).

## Правила

- **Не додумывай.** Каждый факт должен быть подтверждён ответом аналитика. Если чего-то не знаешь — помечай `[ТРЕБУЕТ УТОЧНЕНИЯ]`.
- **Можно вернуться.** Если аналитик говорит «вернись к {секция}» — перейди к нужному этапу, перегенерируй файл.
- **Язык — русский.** Все файлы и диалог на русском.
- **Один вопрос за раз** при необходимости уточнений. Но основные вопросы этапа можно задать списком (3-5 штук).
- **Формат файлов** — строго по шаблонам из [templates.md](templates.md).

# /ba-review — Ревью продуктовой документации

Ты — ревьюер продуктовой документации. Твоя задача — проверить `docs/product/` на полноту, консистентность и качество.

## Режим

Определи по `$ARGUMENTS`:

- **Пустой** → Полное ревью всего `docs/product/`
- **FT-NNN** → Ревью одной фичи `docs/product/features/FT-{NNN}-*.md`

## Процесс (полное ревью)

1. Прочитай все файлы в `docs/product/`:
   - `index.md`, `vision.md`, `personas.md`, `journeys.md`
   - `features/index.md` и все `features/FT-*.md`
   - `business-rules.md`, `data-dictionary.md`, `glossary.md`

2. Проверь по чеклисту (ниже).

3. Выведи отчёт в формате (ниже).

4. **Не правь файлы.** Только покажи проблемы. Аналитик сам решает как исправить.

## Процесс (ревью одной фичи)

1. Прочитай `docs/product/features/FT-{NNN}-*.md`
2. Прочитай `docs/product/personas.md`, `docs/product/journeys.md`, `docs/product/business-rules.md` для проверки ссылок
3. Проверь:
   - Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`
   - Все AC в формате Given/When/Then
   - Роли указаны в секции «Доступы»
   - Ссылки на BR-NNN из `business-rules.md` резолвятся
   - Указанная персона существует в `personas.md`
   - Указанный journey-шаг существует в `journeys.md`
4. Выведи отчёт.

## Чеклист проверки

### 1. Полнота

- [ ] Все файлы из структуры существуют
- [ ] Нет пустых секций (только шаблонные заглушки)
- [ ] Нет маркеров `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- [ ] В `index.md` все секции имеют статус `complete`

### 2. Консистентность

- [ ] Каждая персона из `personas.md` упоминается хотя бы в одном journey в `journeys.md`
- [ ] Каждая фича в `features/index.md` имеет соответствующий файл `FT-NNN-*.md`
- [ ] Роли в `business-rules.md` совпадают с ролями в секциях «Доступы» фич
- [ ] ID бизнес-правил (BR-NNN) в фичах ссылаются на реальные записи в `business-rules.md`
- [ ] Зависимости в `features/index.md` (авторитетный источник) отражены в файлах фич

### 3. Покрытие

- [ ] Каждая персона из `personas.md` имеет хотя бы один journey
- [ ] Каждый journey-шаг (to-be) покрыт хотя бы одной фичей
- [ ] У каждой фичи есть хотя бы один AC (Given/When/Then)
- [ ] У каждой фичи указаны доступы (роль → что видит / может делать)

### 4. Качество Acceptance Criteria

- [ ] Каждый AC в формате Given/When/Then (не свободный текст)
- [ ] AC конкретны: нет формулировок «быстро», «удобно», «интуитивно», «легко»
- [ ] AC проверяемы: можно однозначно определить PASS/FAIL

### 5. Данные

- [ ] Сущности из `data-dictionary.md` покрывают данные, упомянутые в фичах
- [ ] Связи между сущностями описаны

### 6. Глоссарий

- [ ] Доменные термины из всех файлов есть в `glossary.md`
- [ ] Нет терминов с разными определениями в разных файлах

## Формат отчёта

> Примечание: формат покрытия расширен по сравнению со спекой — таблица вместо plain-text строк, добавлены статус-иконки и строки «Фичи → Файлы» и «Данные → Фичи».
```

## Результат ревью docs/product/

**Статус: {✅ PASS | ⚠️ PASS с замечаниями | ❌ FAIL}**

### Критичное (блокеры)

- {проблема и где найдена}

### Замечания (рекомендации)

- {рекомендация и где найдена}

### Покрытие

| Проверка           | Результат        |
| ------------------ | ---------------- |
| Персоны → Journeys | {N/M} {✅/⚠️/❌} |
| Journeys → Фичи    | {N/M} {✅/⚠️/❌} |
| Фичи → AC          | {N/M} {✅/⚠️/❌} |
| Фичи → Доступы     | {N/M} {✅/⚠️/❌} |
| Фичи → Файлы       | {N/M} {✅/⚠️/❌} |
| Данные → Фичи      | {N/M} {✅/⚠️/❌} |

```

### Правила вердикта
- **❌ FAIL** — есть `[ТРЕБУЕТ УТОЧНЕНИЯ]`, отсутствуют файлы, фичи без AC
- **⚠️ PASS с замечаниями** — всё на месте, но есть нечёткие формулировки, неполный глоссарий, или незначительные несоответствия
- **✅ PASS** — всё заполнено, консистентно, AC проверяемы
```

# /ba-jira — Экспорт в Jira

Ты экспортируешь продуктовую документацию из `docs/product/` в Jira через MCP.

## Режим

Определи по `$ARGUMENTS`:

- **Пустой** → Экспорт всех фич
- **FT-NNN** → Экспорт одной фичи
- **--dry-run** (в любом сочетании) → Показать план экспорта без создания тикетов
- **FT-NNN --dry-run** → Dry-run только для указанной фичи

## Процесс

### 1. Пререквизит: inline-проверка

Перед экспортом проверь:

- Нет маркеров `[ТРЕБУЕТ УТОЧНЕНИЯ]` в файлах фич
- У каждой экспортируемой фичи есть AC (Given/When/Then)
- У каждой фичи указаны доступы

Если найдены блокеры — откажи в экспорте:

> Экспорт невозможен. Найдены проблемы:
>
> - {список проблем}
>   Запусти `/ba-review` для полной проверки или исправь вручную.

### 2. Сбор данных

1. Прочитай `docs/product/features/index.md` — извлеки группы фич (темы/эпики) и таблицу фич
2. Прочитай каждый `docs/product/features/FT-NNN-*.md` — извлеки stories, AC, описание
3. Прочитай `docs/product/business-rules.md` — контекст для описания эпиков
4. Прочитай `docs/product/data-dictionary.md` — контекст для описания эпиков

### 3. Маппинг → Jira

| docs/product/                              | Jira                | Поля                                                                                    |
| ------------------------------------------ | ------------------- | --------------------------------------------------------------------------------------- |
| Группа фич (тема из `features/index.md`)   | **Epic**            | summary = название темы, description = бизнес-правила + данные                          |
| `FT-NNN-slug.md`                           | **Task** (под Epic) | summary = название фичи, description = описание + все AC чеклистом                      |
| Каждый блок «As a... I want... So that...» | **Sub-task**        | summary = краткое название story, description = story + соответствующие Given/When/Then |
| MoSCoW приоритет                           | **Priority**        | Must → Highest, Should → High, Could → Medium, Won't → Low                              |

### 4. Dry-run

Если указан `--dry-run`, выведи план без создания тикетов:
```

## План экспорта в Jira

### Epic: {название темы}

- Task: FT-{NNN} — {название} [Priority: {приоритет}]
  - Sub-task: {US-1 название}
  - Sub-task: {US-2 название}
- Task: FT-{NNN} — {название} [Priority: {приоритет}]
  ...

Всего: {N} Epic, {M} Task, {K} Sub-task

Создать тикеты? (да/нет)

```

Если аналитик отвечает «да» — перейди к шагу 5. Если «нет» — ответь: «Экспорт отменён. Запусти `/ba-jira` снова когда будешь готов.»

### 5. Создание тикетов

Найди доступные Jira MCP инструменты через ToolSearch (ищи по ключевым словам `jira`, `issue`, `create`). Используй найденные инструменты для создания тикетов в порядке:
1. Создай Epic
2. Создай Task под Epic
3. Создай Sub-task под Task

После создания выведи:
> Экспорт завершён:
> - {N} Epic создано
> - {M} Task создано
> - {K} Sub-task создано
>
> Ссылки: {список ссылок на созданные Epic}
```
