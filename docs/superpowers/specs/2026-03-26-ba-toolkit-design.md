# BA Toolkit — Design Spec

## Суть

Набор скиллов и структурированная директория `docs/product/` для бизнес-аналитика. Аналитик клонирует репу, запускает скилл, через диалог описывает продукт/фичу — на выходе папка с MD-файлами, готовая для передачи разработчику или агентам.

## Два сценария

### Сценарий 1: Новый проект

```
/ba              → docs/product/ (полные 8 этапов, диалог с аналитиком)
/ba-review       → проверка полноты и консистентности ✅
/ralph           → prd.json для автономной разработки агентами
```

> **Примечание:** `/ralph` — отдельный скилл (не часть этого тулкита), конвертирует PRD в `prd.json` для системы Ralph. Его нужно адаптировать для приёма `docs/product/` как входа — см. секцию «Адаптация `/ralph`». `/ralph` генерит машиночитаемый JSON для агентов, в отличие от `/prd`, который генерит human-readable пакет документов.

### Сценарий 2: Существующий проект, новая фича

```
/ba фича         → docs/product/ (этапы 4-5-6-7 — фичи, правила, данные, глоссарий)
/ba-review       → проверка ✅
/ba-jira         → Epic + Tasks в Jira через MCP
```

> **Пререквизит сценария 2:** `personas.md` и `journeys.md` должны существовать и быть заполнены (ссылки на персоны и journey-шаги используются в фичах). Если файлов нет — скилл предложит: «Запусти `/ba` для нового проекта или заполни `personas.md` и `journeys.md` вручную».

## Структура `docs/product/`

```
docs/product/
├── index.md                 # TOC, статус готовности секций, ссылки
├── vision.md                # Проблема, решение, ценность, аудитория, конкуренты, метрики
├── personas.md              # Стейкхолдеры (таблица) + персоны (карточки)
├── journeys.md              # User journeys as-is / to-be по персонам
├── features/
│   ├── index.md             # Карта фич: ID, название, персона, MoSCoW, сложность, зависимости
│   ├── FT-001-slug.md       # Спек фичи
│   └── ...
├── business-rules.md        # BR-NNN: правила домена, роли, доступы
├── data-dictionary.md       # Сущности, атрибуты, связи (бизнес-уровень)
└── glossary.md              # Термины домена
```

### Шаблоны файлов

#### `index.md`

- Название продукта
- Суть в одном предложении
- Таблица готовности секций со статусами: `draft` | `in-progress` | `complete`
  - `draft` — файл создан, но не заполнен
  - `in-progress` — файл частично заполнен или содержит `[ТРЕБУЕТ УТОЧНЕНИЯ]`
  - `complete` — файл заполнен, аналитик подтвердил (скилл ставит после «Ок? Идём дальше»)
- Аннотированные ссылки на все файлы

#### `vision.md`

- Проблема (кому больно, почему)
- Решение (что делаем, в чём отличие)
- Ценностное предложение
- Целевая аудитория (сегменты)
- Конкурентный ландшафт (таблица сравнения)
- Метрики успеха (North Star + 3-5 KPI)
- Ограничения и допущения

#### `personas.md`

- Стейкхолдеры (таблица: роль, интерес, влияние)
- Персоны (карточки): имя, роль, демография, цели, боли, мотивация, текущее поведение, ожидания от продукта

#### `journeys.md`

- Для каждой ключевой персоны:
  - Journey as-is (как сейчас)
  - Journey to-be (как будет с продуктом)
  - Шаги: действие → мысли → боли → возможности
  - Touchpoints с продуктом

#### `features/index.md`

- Карта фич (таблица): ID (FT-001), название, персона, приоритет (MoSCoW), сложность (S/M/L/XL), статус
- Группировка по эпикам/темам
- Граф зависимостей (текстом) — **авторитетный источник** зависимостей; поле «Зависимости» в `FT-NNN` файлах — производное

#### `features/FT-NNN-slug.md`

- Название, описание
- Связанная персона и journey-шаг
- User stories (As a... I want... So that...)
- Acceptance criteria (Given/When/Then)
- Описание UI/UX (экраны, взаимодействия — текстом)
- Edge cases и ошибки
- Доступы (роль → что видит / может делать)
- Бизнес-правила (ссылки на business-rules.md)
- Зависимости от других фич
- Приоритет и обоснование

#### `business-rules.md`

- BR-NNN: правило (если X, то Y)
- Контекст, исключения
- Роли и уровни доступа
- Связанные фичи

#### `data-dictionary.md`

- Сущности (таблица: имя, описание, ключевые атрибуты)
- Связи между сущностями (1:N, N:M)
- Бизнес-ограничения на данные

#### `glossary.md`

- Термин → определение (таблица)
- Синонимы, контекст использования

## Скиллы

### `/ba` — линейный проводник

**Тип:** user-invocable, линейный диалог

**Аргументы:**

- `/ba` — запуск с нуля (все 8 этапов)
- `/ba фича` — режим добавления фичи (этапы 4-5-6-7)
- `/ba продолжить` — читает `index.md`, находит первую секцию со статусом `draft` или `in-progress`, продолжает с неё. Если `index.md` не существует — предлагает запустить `/ba` с нуля

**Этапы:**

| #   | Этап           | Вопросы                                                          | Результат                        |
| --- | -------------- | ---------------------------------------------------------------- | -------------------------------- |
| 1   | Идея           | Какая проблема? Для кого? Чем отличается? Как измерить успех?    | `vision.md`                      |
| 2   | Персоны        | Кто пользователи? Роли, боли, цели? Кто стейкхолдеры?            | `personas.md`                    |
| 3   | Journeys       | Как пользователь решает проблему сейчас? Как будет? Touchpoints? | `journeys.md`                    |
| 4   | Фичи           | Какие возможности нужны? Что must/should/could/won't?            | `features/index.md`              |
| 5   | Спеки фич      | По каждой фиче: stories, AC, UI, доступы, edge cases             | `features/FT-NNN-slug.md` (цикл) |
| 6   | Бизнес-правила | Какие правила домена? Роли и доступы? Исключения?                | `business-rules.md`              |
| 7   | Данные         | Какие сущности? Атрибуты? Связи? Ограничения?                    | `data-dictionary.md`             |
| 8   | Глоссарий      | Скилл парсит термины из файлов, аналитик дополняет               | `glossary.md` + `index.md`       |

**Правила:**

- Каждый факт подтверждён ответом аналитика — не додумывать
- Неизвестное помечать `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- Можно вернуться к любому этапу по просьбе
- Этап 5 циклический — аналитик может пропустить фичу («детализируем позже»), пропущенная фича остаётся со статусом `draft`
- Глоссарий полуавтоматический — скилл предлагает термины, аналитик правит
- 3-5 вопросов на этап (открытые — аналитик думает своими словами)
- После генерации файла: «Ок? Правим или идём дальше?»

**Финал:** после этапа 8 выводит:

> Документация готова в `docs/product/`. Запусти `/ba-review` для проверки полноты.
>
> - Новый проект → `/ralph` для автономной разработки
> - Существующий проект → `/ba-jira` для создания задач в Jira

### `/ba-review` — ревьюер

**Тип:** user-invocable

**Аргументы:**

- `/ba-review` — полное ревью всего `docs/product/`
- `/ba-review FT-003` — ревью одной фичи (проверяет: нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`, все AC в формате Given/When/Then, роли указаны, ссылки на business-rules резолвятся, персона и journey-шаг существуют)

**Чеклист проверки:**

| Категория       | Что проверяет                                                                |
| --------------- | ---------------------------------------------------------------------------- |
| Полнота         | Все файлы существуют? Нет пустых секций? Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`?          |
| Консистентность | Персоны → journeys → фичи → правила — цепочка не рвётся?                     |
| Покрытие        | Каждая персона в journey? Каждый journey-шаг покрыт фичей? У каждой фичи AC? |
| Качество AC     | Конкретные и проверяемые? Нет размытых формулировок?                         |
| Доступы         | У каждой фичи указаны роли? Совпадают с `business-rules.md`?                 |
| Данные          | Сущности покрывают данные из фич?                                            |
| Глоссарий       | Термины из доков есть в `glossary.md`?                                       |

**Формат вывода:**

```
## Результат ревью

Статус: ✅ PASS / ⚠️ PASS с замечаниями / ❌ FAIL

### Критичное (блокеры)
- ...

### Замечания (рекомендации)
- ...

### Покрытие
Персоны → Journeys: N/N
Journeys → Фичи: N/N
Фичи → AC: N/N
Фичи → Доступы: N/N
```

**Не правит файлы сам** — только показывает проблемы.

### `/ba-jira` — экспорт в Jira

**Тип:** user-invocable

**Аргументы:**

- `/ba-jira` — экспорт всех фич
- `/ba-jira FT-003` — экспорт одной фичи
- `/ba-jira --dry-run` — показать что будет создано, без записи

**Что делает:**

1. Читает `docs/product/features/index.md` — группы фич (→ Jira Epic)
2. Читает `docs/product/features/FT-NNN-slug.md` — каждая фича (→ Jira Task под Epic)
3. Каждый блок «As a... I want... So that...» → отдельная Sub-task, с соответствующими Given/When/Then в описании
4. Из `business-rules.md` и `data-dictionary.md` → описание Epic
5. Создаёт через Jira MCP

**Маппинг:**

| docs/product/                                 | Jira                                                         |
| --------------------------------------------- | ------------------------------------------------------------ |
| Группа фич (тема/эпик из `features/index.md`) | Epic                                                         |
| `FT-NNN-slug.md`                              | Task (под Epic)                                              |
| User story из FT-NNN (каждый «As a...» блок)  | Sub-task (с AC в описании)                                   |
| MoSCoW приоритет                              | Priority: Must→Highest, Should→High, Could→Medium, Won't→Low |
| Acceptance criteria                           | Description задачи (чеклист)                                 |

**Пререквизит:** перед экспортом выполняет inline-проверку (тот же чеклист что и `/ba-review`). Если найдены блокеры (`[ТРЕБУЕТ УТОЧНЕНИЯ]` или отсутствующие AC) — отказывает в экспорте и показывает список проблем.

---

### Адаптация `/ralph` (вне скоупа этого тулкита)

Скилл `/ralph` конвертирует PRD в машиночитаемый `prd.json` для автономной системы Ralph. Это отличается от `/prd`, который генерит human-readable пакет документов для разработчиков.

Адаптация: научить `/ralph` принимать `docs/product/` как вход:

- `/ralph docs/product/` → читает всю структуру → генерит `prd.json`
- Маппинг: vision → product description, features → tasks, AC → acceptance criteria, data-dictionary → data model

> Эта адаптация выполняется отдельно, не является частью BA-тулкита.

---

## Что НЕ входит в BA-тулкит

- Технические спецификации (зона разработчика)
- NFR: производительность, безопасность в техническом смысле (кроме ролей/доступов)
- Архитектурные решения
- Технический durable-docs слой — не используется в цепочке аналитика
