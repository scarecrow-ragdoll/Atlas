# Atlas PRD

## 1. Назначение проекта

Atlas — self-hosted веб-приложение для ведения персонального дневника тренировок, питания, замеров тела, фото прогресса и подготовки структурированной выгрузки для AI-анализа.

Главная идея Atlas — не просто хранить тренировочные данные, а автоматизировать еженедельный цикл:

1. В течение недели пользователь вносит тренировки, кардио, вес и питание.
2. Раз в неделю делает body check-in: вес, обхваты, фото.
3. Приложение собирает данные за выбранный период, по умолчанию за последние 4 недели.
4. Приложение формирует AI prompt и файл с данными.
5. Пользователь отправляет prompt и файл в ChatGPT/другой AI.
6. AI анализирует прогресс и даёт рекомендации на следующую неделю.

## 2. Цель продукта

Создать персональную систему отслеживания фитнес-прогресса, которая позволяет:

* быстро вносить данные по тренировкам;
* отслеживать прогрессию рабочих весов;
* учитывать кардио и общую нагрузку;
* отслеживать вес, обхваты тела и фото;
* вести простой дневник питания через недельный шаблон;
* строить графики прогресса;
* экспортировать данные для AI-анализа;
* делать полный локальный backup/restore данных.

## 6. Основные продуктовые принципы

### 6.1 Минимум рутины

Приложение должно помогать вносить данные максимально быстро.

Ключевые требования:

* по умолчанию открыт текущий день;
* можно выбрать любую дату через календарь;
* можно вносить данные задним числом;
* поля должны быть предзаполнены там, где это возможно;
* рабочий вес упражнения должен подставляться автоматически;
* пользователь не должен каждый день вручную вбивать питание, если оно соответствует недельному шаблону.

### 6.2 Данные должны быть полезны для AI

Все основные сущности должны проектироваться так, чтобы их можно было использовать в AI-выгрузке.

AI должен получать:

* тренировки;
* упражнения;
* подходы;
* веса;
* повторения;
* рабочие веса;
* комментарии пользователя;
* RPE/RIR, если указаны;
* кардио;
* вес тела;
* обхваты;
* фото, если пользователь включил их в экспорт;
* питание;
* цель пользователя;
* дополнительный контекст.

### 6.3 Данные принадлежат пользователю

Пользователь должен иметь возможность:

* экспортировать все данные;
* сохранить локальный backup;
* развернуть новый экземпляр приложения;
* импортировать backup;
* восстановить данные без ручной миграции.

## 7. Пользователь и доступ

## 7.1 Пользователь

В MVP пользователь один.

Не требуется:

* регистрация;
* управление пользователями;
* роли;
* приглашения;
* публичные профили.

## 7.2 PIN-код

В приложении должна быть возможность включить PIN-код.

PIN-код нужен, чтобы данные не были доступны всем, кто открыл URL приложения.

Требования:

* PIN-код включается опционально;
* если PIN-код выключен, приложение открывается без проверки;
* если PIN-код включен, при открытии приложения пользователь должен ввести PIN;
* PIN не должен храниться в открытом виде;
* должна быть возможность изменить PIN;
* должна быть возможность отключить PIN;
* сессия после ввода PIN должна сохраняться через cookie/session;
* чувствительные данные не должны быть доступны без валидной сессии.

## 10. Дневник тренировок

## 10.1 Общая модель

Тренировка в Atlas — это дневная запись активности.

Все упражнения, внесённые за определённую дату, считаются выполненными в этот день и входят в одну тренировочную запись.

Пример:

* утром пользователь сделал кардио;
* вечером сделал силовую тренировку;
* в системе это хранится как активность за одну дату.

## 10.2 Работа с датой

Требования:

* по умолчанию открыт текущий день;
* пользователь может выбрать дату через календарь;
* можно добавлять и редактировать тренировки задним числом;
* если за дату уже есть запись, открывается существующая запись;
* если за дату записи нет, создаётся новая запись при первом сохранении данных.

## 10.3 Что хранится в тренировочном дне

Тренировочный день должен включать:

* дату;
* список упражнений;
* подходы по каждому упражнению;
* кардио за этот день;
* общий комментарий к дню;
* опциональный вес тела на этот день.

## 10.4 Упражнение внутри тренировочного дня

Для каждого упражнения в тренировочном дне хранится:

* ссылка на упражнение из справочника;
* порядок упражнения в списке;
* рабочий вес на момент выполнения;
* комментарий пользователя к этому упражнению;
* список подходов.

Комментарий к упражнению должен попадать в AI-выгрузку.

Примеры комментариев:

* последний подход был отказным;
* из-за боли в груди не смог выполнить план;
* вес ощущался легко;
* техника была хуже обычного;
* нужно повторить этот вес на следующей неделе.

## 10.5 Подходы

Каждый подход должен включать:

* номер подхода;
* вес;
* количество повторений;
* опционально RPE;
* опционально RIR;
* опционально комментарий.

RPE/RIR не обязательны.

Если пользователь не хочет их указывать, он может вести тренировку только через вес и повторы.

## 10.6 Рабочий вес

У каждого упражнения должен быть рабочий вес.

Рабочий вес — это базовый ориентир, от которого пользователь отталкивается при выполнении упражнения.

Требования:

* рабочий вес хранится в справочнике упражнения;
* при добавлении упражнения в тренировочный день рабочий вес подставляется автоматически;
* пользователь может выполнить упражнение с большим или меньшим весом;
* фактический вес подходов хранится отдельно;
* в тренировочном дне сохраняется snapshot рабочего веса на момент выполнения;
* AI-выгрузка должна включать рабочий вес и фактические веса;
* AI должен иметь возможность рекомендовать повышение/понижение рабочих весов.

## 10.7 Прогрессия

Приложение должно помогать отслеживать прогрессию.

MVP должен уметь показывать:

* текущий рабочий вес;
* фактические веса за выбранный период;
* лучший подход;
* объём по упражнению;
* estimated 1RM;
* динамику по неделям.

Приложение должно отмечать потенциальные сигналы прогрессии:

* пользователь стабильно выполняет верхнюю границу повторений;
* вес растёт;
* объём растёт;
* вес стоит на месте;
* результат ухудшается;
* пользователь часто пишет негативные комментарии по упражнению.

Автоматическое изменение рабочего веса без подтверждения пользователя не требуется.

## 10.8 Не входит в MVP

В MVP не нужны:

* шаблоны тренировок;
* планирование тренировок заранее;
* быстрое повторение прошлой тренировки;
* тренировочные программы;
* разделение тренировок по сплитам;
* календарь будущих тренировок.

## 11. Справочник упражнений

## 11.1 Создание упражнений

Пользователь сам создаёт упражнения.

В MVP не требуется стартовый каталог упражнений.

## 11.2 Поля упражнения

Упражнение должно включать:

* название;
* основные мышечные группы;
* описание;
* личные заметки;
* рабочий вес;
* изображения;
* видео;
* статус активности.

## 11.3 Медиа

Пользователь должен иметь возможность сам загружать изображения и видео.

Требования:

* изображения и видео привязаны к упражнению;
* медиа можно добавлять после создания упражнения;
* медиа можно удалить;
* медиа должны попадать в полный backup export;
* медиа не обязательно включать в AI export.

## 11.4 Стандартные и пользовательские упражнения

В будущем может появиться стартовый набор упражнений.

Если он будет добавлен, стандартные упражнения и упражнения пользователя не должны отличаться в основной логике приложения.

Пользователь должен иметь возможность:

* редактировать стандартное упражнение для себя;
* добавить свои заметки;
* поменять рабочий вес;
* использовать упражнение в тренировке так же, как пользовательское.

## 12. Кардио

## 12.1 Назначение

Кардио — дополнительный контекст для анализа нагрузки, восстановления и прогресса.

## 12.2 Поля кардио

Кардио-запись должна включать:

* дату;
* тип кардио;
* длительность в минутах;
* пульс;
* зона пульса;
* комментарий.

## 12.3 Типы кардио

В MVP пользователь должен иметь возможность выбрать тип кардио.

Базовые типы:

* ходьба;
* бег;
* велотренажёр;
* эллипс;
* дорожка;
* другое.

## 12.4 Пульс и зоны

Кардио должно учитывать пульс/зону пульса.

Минимальный вариант:

* средний пульс;
* зона пульса.

Зоны можно задавать как enum:

* Zone 1;
* Zone 2;
* Zone 3;
* Zone 4;
* Zone 5;
* unknown.

Если пользователь не знает зону, он может указать только пульс или оставить поле пустым.

## 13. Замеры тела

## 13.1 Общая логика

Пользователь может вносить замеры тела раз в неделю или чаще.

Основной сценарий — weekly check-in.

Дополнительный сценарий — пользователь может вносить вес чаще, чтобы в AI-отчёте была видна динамика веса за неделю.

## 13.2 Weekly check-in

Weekly check-in должен включать:

* дату;
* вес;
* процент жира опционально;
* фотографии 2-4 штуки;
* обхваты тела;
* комментарий.

## 13.3 Обхваты тела

Список обхватов:

* шея;
* плечи;
* предплечья;
* бицепс;
* грудь;
* талия;
* живот;
* таз;
* бедро;
* икры.

## 13.4 Левая и правая сторона

Для парных замеров должна быть возможность указать левую и правую сторону.

Парные замеры:

* предплечья;
* бицепс;
* бедро;
* икры.

Правило:

* если пользователь указал только одно значение, оно считается общим;
* если пользователь указал левое и правое значение, сохраняются оба;
* если второе значение не указано, приложение не должно требовать его обязательного заполнения.

## 13.5 Вес тела

Вес тела можно вносить чаще, чем раз в неделю.

Требования:

* можно добавить вес на любую дату;
* вес может быть частью weekly check-in;
* вес может быть отдельной записью;
* графики должны показывать динамику веса за выбранный период;
* AI export должен включать все записи веса за выбранный период.

## 14. Фото прогресса

## 14.1 Привязка фото

Фото прогресса привязываются к weekly check-in.

Каждое фото должно иметь:

* дату;
* связь с check-in;
* файл;
* опциональную метку;
* опциональный угол съёмки;
* опциональный комментарий.

## 14.2 Углы фото

Базовые углы:

* front;
* side;
* back;
* custom.

## 14.3 Хранение фото

Конкретный способ хранения не принципиален для пользователя.

Требования продукта:

* пользователь должен иметь возможность открыть фото в любой момент;
* фото не должны быть публично доступны без авторизации/PIN-сессии;
* фото должны включаться в полный backup export;
* фото должны включаться в AI export только если пользователь явно выбрал эту опцию.

## 15. Питание

## 15.1 Назначение

Питание в Atlas не должно быть полноценным ежедневным calorie tracker.

Основная идея — недельный шаблон питания.

Пользователь раз в неделю задаёт типовой день питания, а приложение считает, что каждый день этой недели пользователь ел по этому шаблону.

Если в конкретный день были отличия, пользователь может внести override.

## 15.2 Справочник продуктов

Пользователь вручную создаёт продукты.

Каждый продукт включает:

* название;
* калории на 100 г;
* белки на 100 г;
* жиры на 100 г;
* углеводы на 100 г;
* комментарий опционально.

## 15.3 Недельный шаблон питания

Пользователь создаёт один шаблон питания на неделю.

Шаблон включает список продуктов и количество граммов в день.

Каждая строка шаблона включает:

* продукт;
* количество граммов;
* опциональный приём пищи;
* опциональный комментарий.

Приложение должно считать:

* калории;
* белки;
* жиры;
* углеводы.

## 15.4 Применение шаблона

По умолчанию каждый день недели использует недельный шаблон.

Пример:

* пользователь в воскресенье создал шаблон;
* приложение считает, что с понедельника по воскресенье питание соответствует шаблону;
* если пользователь в среду съел что-то дополнительно, он редактирует только среду.

## 15.5 Daily override

Для конкретного дня пользователь может:

* добавить продукт;
* убрать продукт;
* изменить количество продукта;
* оставить комментарий.

Daily override должен влиять только на выбранную дату.

## 15.6 Не входит в MVP

В MVP не нужно:

* рецепты;
* готовые блюда;
* barcode scanner;
* учёт воды;
* учёт клетчатки;
* учёт соли;
* учёт сахара;
* учёт алкоголя;
* автоматическое распознавание еды;
* публичная база продуктов.

## 16. Графики

## 16.1 Общие требования

Пользователь должен видеть графики с динамикой изменения за выбранный период.

Требования:

* пользователь выбирает период;
* данные можно фильтровать;
* графики должны быть простыми и понятными;
* графики должны использовать данные из тренировок, замеров и питания.

## 16.2 Графики тренировок

Для упражнения должны быть доступны графики:

* рабочий вес;
* лучший подход;
* estimated 1RM;
* общий объём;
* общее количество повторений;
* количество рабочих подходов.

Пользователь должен выбрать конкретное упражнение и период.

## 16.3 Графики тела

Должны быть доступны графики:

* вес тела;
* процент жира;
* каждый отдельный обхват;
* несколько выбранных обхватов на одном графике.

Примеры:

* вес за 4 недели;
* талия за 3 месяца;
* бицепс и бедро за выбранный период;
* грудь, талия, живот за период сушки.

## 16.4 Графики питания

Питание в MVP требует только базовых графиков.

Минимальные графики:

* средние калории по неделям;
* средний белок по неделям;
* средние жиры по неделям;
* средние углеводы по неделям.

## 17. AI Export

## 17.1 Назначение

AI Export — одна из ключевых функций Atlas.

Пользователь должен получить:

* готовый prompt для AI;
* файл с данными за выбранный период;
* человекочитаемую сводку;
* структурированные данные для анализа.

## 17.2 Период

Требования:

* период настраиваемый;
* период по умолчанию — последние 4 недели;
* пользователь может выбрать произвольную дату начала и конца.

## 17.3 Состав экспорта

Пользователь может выбрать, что включить:

* тренировки;
* упражнения;
* подходы;
* рабочие веса;
* комментарии;
* RPE/RIR;
* кардио;
* вес тела;
* обхваты;
* фото;
* питание;
* цель пользователя;
* дополнительный контекст.

Фото не должны включаться по умолчанию.

## 18. AI Prompt Builder

## 18.1 Назначение

Prompt Builder должен сформировать готовый текст, который пользователь отправит в ChatGPT вместе с файлом экспорта.

## 18.2 Постоянный контекст

Приложение должно хранить постоянный AI-контекст.

Поля:

* текущая цель;
* рост;
* возраст опционально;
* тренировочный стаж;
* текущий тренировочный сплит опционально;
* ограничения/травмы;
* предпочитаемый стиль прогрессии;
* текущая стратегия питания;
* дополнительный постоянный комментарий.

Пользователь может изменить цель и контекст в любой момент.

Если цель введена один раз, приложение должно запомнить её для следующих экспортов.

## 18.3 Разовый контекст

При генерации AI export пользователь может добавить разовый комментарий.

Примеры:

* сейчас нахожусь на курсе AAS;
* неделя была тяжёлая по сну;
* был высокий стресс;
* пропустил тренировку;
* болело плечо;
* питание отличалось от шаблона;
* хочу сделать акцент на силовой прогресс;
* хочу снизить нагрузку.

Разовый комментарий должен попадать только в конкретный AI export.

## 18.4 Флаги недели

Приложение должно поддерживать флаги недели.

Базовые флаги:

* плохой сон;
* высокий стресс;
* болезнь;
* травма/боль;
* AAS/cycle context;
* дефицит калорий;
* профицит калорий;
* поддержание;
* пропущенные тренировки;
* путешествие/сбитый режим.

Флаги должны попадать в AI prompt и `data.json`.

## 18.5 Что prompt должен просить у AI

Prompt должен просить AI:

* проанализировать прогресс по упражнениям;
* оценить динамику рабочих весов;
* сравнить фактические веса с рабочими;
* оценить объём нагрузки;
* учесть RPE/RIR и комментарии;
* учесть кардио;
* сравнить тренировки с динамикой веса и обхватов;
* учесть питание;
* дать рекомендации на следующую неделю;
* предложить изменения рабочих весов;
* отметить упражнения, где стоит повысить вес;
* отметить упражнения, где лучше повторить вес;
* отметить упражнения, где возможна перегрузка или откат;
* дать практический план действий.

## 19. AI Review History

## 19.1 Назначение

Приложение должно иметь возможность сохранять историю AI-рекомендаций.

Это полезно, чтобы видеть:

* что AI советовал;
* какие рекомендации были применены;
* какой был результат;
* как менялись решения по тренировкам.

## 19.2 MVP-уровень

В MVP достаточно простой реализации:

* пользователь может создать AI review;
* вставить текст ответа от AI;
* привязать его к периоду;
* добавить комментарий;
* отметить, какие рекомендации он планирует применить.

Автоматическая интеграция с OpenAI/ChatGPT в MVP не требуется.

## 20. Import / Export

## 20.1 Full backup export

Atlas должен поддерживать полный экспорт всех данных.

Назначение:

* локальный backup;
* перенос данных;
* восстановление после потери инстанса;
* миграция на новый сервер.

Формат: ZIP archive.

Структура:

```text
atlas-backup-YYYY-MM-DD.zip
  manifest.json
  data.json
  media/
```

Опционально можно добавить CSV-файлы для ручной проверки:

```text
  csv/
    exercises.csv
    workouts.csv
    sets.csv
    body_measurements.csv
    nutrition.csv
```

## 20.2 manifest.json

`manifest.json` должен включать:

* тип экспорта: full_backup;
* schemaVersion;
* appVersion;
* exportedAt;
* includedSections;
* mediaIncluded;
* file list;
* optional checksums.

## 20.3 data.json

`data.json` должен включать все основные сущности:

* settings;
* user profile;
* goals;
* exercises;
* exercise media;
* workout days;
* workout exercises;
* workout sets;
* cardio entries;
* body weight entries;
* body check-ins;
* body measurements;
* progress photos metadata;
* nutrition products;
* nutrition templates;
* daily nutrition overrides;
* AI prompt settings;
* AI review history.

## 20.4 Full import

Импорт должен позволять восстановить данные в новом экземпляре приложения.

Требования:

* загрузка ZIP-файла;
* проверка `manifest.json`;
* проверка версии схемы;
* проверка структуры `data.json`;
* dry-run validation перед импортом;
* отображение summary перед импортом;
* понятные ошибки;
* запрет silent partial import;
* восстановление media-файлов;
* восстановление связей между сущностями.

## 20.5 Версионирование схемы

Формат экспорта должен иметь версию схемы.

Пример:

```json
{
  "schemaVersion": "1.0.0"
}
```

Будущие версии приложения должны учитывать миграции backup-формата.

## 21. Оптимизация рутины

Atlas должен помогать пользователю делать меньше ручных действий.

В MVP нужно реализовать:

* открытие текущего дня по умолчанию;
* выбор даты через календарь;
* ввод данных задним числом;
* автоподстановку рабочего веса;
* быстрое добавление подходов;
* копирование значения прошлого подхода внутри упражнения;
* подсчёт тоннажа;
* подсчёт estimated 1RM;
* недельный шаблон питания;
* daily overrides вместо ежедневного полного ввода питания;
* флаги недели;
* сохранение цели пользователя;
* генерацию prompt;
* генерацию ZIP export.

Не требуется в MVP:

* быстрое повторение прошлой тренировки.

## 22. Apple Health

## 22.1 Статус

Apple Health integration не входит в MVP.

Причина: Atlas — веб-приложение, а полноценная интеграция с Apple Health требует отдельного Apple ecosystem bridge, например iOS-приложения или другого механизма импорта данных.

## 22.2 Future scope

В будущем можно рассмотреть:

* отдельное iOS-приложение-компаньон;
* ручной импорт Apple Health export;
* импорт веса;
* импорт тренировок;
* импорт шагов;
* импорт активности;
* импорт пульса.

Для MVP вместо Apple Health достаточно ручного ввода веса, кардио и активности.

## 23. Telegram Bot

В стеке указан `go-telegram/bot`, но Telegram bot не входит в MVP Atlas.

Возможное future scope:

* быстро добавить вес через Telegram;

## 24. Нефункциональные требования

## 24.1 Privacy

Требования:

* не логировать PIN;
* не логировать содержимое AI export;
* не логировать фото;
* не логировать чувствительные комментарии;
* не отдавать media-файлы без авторизации/PIN-сессии;

* import должен проверять целостность архива.

## 24.3 Производительность

MVP рассчитан на одного пользователя.

Ожидаемый объём данных:

* тысячи подходов;
* сотни фото;
* сотни продуктов/записей питания.

Интерфейс должен оставаться быстрым на персональном объёме данных.

## 24.4 Тестирование

Критичные сценарии должны быть покрыты тестами:

## 26. Основные пользовательские сценарии

## 26.1 Добавить упражнение

1. Пользователь открывает справочник упражнений.
2. Нажимает “Добавить упражнение”.
3. Вводит название.
4. Указывает мышечные группы.
5. Указывает рабочий вес.
6. Добавляет описание и личные заметки.
7. Загружает изображения/видео опционально.
8. Сохраняет упражнение.

## 26.2 Внести тренировку за сегодня

1. Пользователь открывает дневник тренировок.
2. По умолчанию открыта сегодняшняя дата.
3. Пользователь добавляет упражнение.
4. Система подставляет рабочий вес.
5. Пользователь добавляет подходы.
6. Для каждого подхода указывает вес и повторы.
7. RPE/RIR указывает опционально.
8. Добавляет комментарий к упражнению опционально.
9. Сохраняет день.

## 26.3 Внести тренировку задним числом

1. Пользователь открывает дневник.
2. Через календарь выбирает прошлую дату.
3. Если запись есть, она открывается.
4. Если записи нет, создаётся новая запись.
5. Пользователь вносит упражнения, подходы, кардио и комментарии.
6. Сохраняет данные.

## 26.4 Добавить кардио

1. Пользователь открывает текущий день или выбирает дату.
2. Добавляет кардио.
3. Выбирает тип.
4. Указывает длительность.
5. Указывает пульс и/или зону пульса.
6. Добавляет комментарий опционально.
7. Сохраняет.

## 26.5 Сделать weekly check-in

1. Пользователь открывает раздел замеров.
2. Создаёт check-in.
3. Указывает дату.
4. Указывает вес.
5. Указывает процент жира опционально.
6. Указывает обхваты.
7. Добавляет 2-4 фото.
8. Добавляет комментарий.
9. Сохраняет.

## 26.6 Добавить вес отдельно

1. Пользователь открывает раздел веса или dashboard.
2. Добавляет вес на выбранную дату.
3. Сохраняет.
4. Запись попадает в график веса и AI export.

## 26.7 Создать недельный шаблон питания

1. Пользователь открывает питание.
2. Создаёт продукты в справочнике.
3. Создаёт шаблон на неделю.
4. Добавляет продукты и граммовки.
5. Система считает КБЖУ.
6. Пользователь подгоняет значения под цель.
7. Сохраняет шаблон.
8. Шаблон применяется ко всем дням недели.

## 26.8 Изменить питание за конкретный день

1. Пользователь выбирает дату.
2. Видит питание, рассчитанное из шаблона.
3. Добавляет, убирает или меняет продукт.
4. Система пересчитывает КБЖУ за день.
5. Изменения применяются только к выбранной дате.

## 26.9 Сформировать AI report

1. Пользователь открывает AI Export.
2. Выбирает период.
3. По умолчанию выбран период 4 недели.
4. Выбирает, какие данные включить.
5. Добавляет разовый комментарий.
6. Проверяет сохранённую цель.
7. Система формирует prompt.
8. Система формирует ZIP export.
9. Пользователь скачивает файл.
10. Пользователь отправляет prompt и файл в ChatGPT.

## 26.10 Сохранить AI review

1. Пользователь получает ответ от AI.
2. Открывает AI Review.
3. Вставляет ответ.
4. Привязывает его к периоду.
5. Добавляет свои заметки.
6. Отмечает planned actions.
7. Сохраняет.

## 26.11 Сделать полный backup

1. Пользователь открывает Import/Export.
2. Нажимает Export all data.
3. Выбирает, включать ли media.
4. Система создаёт ZIP backup.
5. Пользователь сохраняет файл локально.

## 26.12 Восстановить данные из backup

1. Пользователь разворачивает новый экземпляр Atlas.
2. Открывает Import/Export.
3. Загружает ZIP backup.
4. Система делает dry-run validation.
5. Показывает summary импорта.
6. Пользователь подтверждает импорт.
7. Система восстанавливает данные и media.
